# Elements of Programming

Every programming language has 3 mechanisms for combining **SIMPLE** ideas to form more **COMPLEX** ones.

1. Primitive expressions
2. Means of combination
3. Means of abstraction

There are two kinds of elements: 
	1. Data - stuff we manipulate
	2. Procedures - descriptions of rules for manipulating DATA


1.1.1 Expressions

486	Primitive expression
+	Primitive procedure
(+ 1 2)	Compound expression

(+                 1               2) is called a combination. Value: apply procedure (operator) to arguments (value of operands)
 operator     operands

Interpreter runs in a read-eval-print loop.

1.1.2 Naming and the Environment

Programming languages need to provide a way to use names to refer to a computational object.

(define size 2) <-- Scheme's way to name objects, also SIMPLEST mean of abstraction

The environment is a piece of memory that tracks name-object pairs.


1.1.3 Evaluating Combinations

GENERAL EVALUATION RULE
	1. Evaluate the subexpressions of the combination.
	2. Apply procedure to the arguments.

Tree accumulation is the process of manipulating data within in a tree-like structure.

Rules for evaluating primitive expressions:
	1. the values of numerals are the numbers that they name,
	2. the values of built-in operators are the machine instruction sequences that carry out the corresponding operations, and
	3. the values of other names are the objects associated with those names in the environment.

Exceptions to the general evaluation rule are called special forms, which have their own evaluation rules.
	- Ex: define

1.1.4 Compound Procedures

Procedure definitions is an abstraction technique by which a compound operation be given a name and can be referred to as a unit.

Ex: (define ( square x ) ( * x x ))    <----  compound procedure, 1. creates procedure 2. gives it a name

General Procedure Definition Form
		(define (⟨name⟩ ⟨formal parameters⟩) ⟨body⟩)
		
Compound procedures are used in exactly the same way as primitive procedures.

1.1.6 Conditional Expressions and Predicates

Case analysis is the construct of making tests and performing different operations 
based on the result of a test.

A special form, cond, is used to indicate a case analysis.
The general form for cond:
					(cond (⟨p₁⟩ ⟨e₁⟩)
      (⟨p₂⟩ ⟨e₂⟩)
      …
      (⟨pₙ⟩ ⟨eₙ⟩))  <-- clauses
					
			Def: <p> = predicate | <e> =  consequent expression

A predicate is an expression whose value is either true or false. 

A special form, if, is a conditional expression with only 2 cases.
The general form for if:
				(if ⟨predicate⟩ ⟨consequent⟩ ⟨alternative⟩)
	
Note: and, or are special forms. not is a regular procedure.









1.1.5 The Substitution Model for Procedure Application

Evaluating a combination with a compound procedure is similar to one with a primitive procedure.

Here is the difference:
	- Applying primitive procedures to arguments is built into interpreter.
	- Applying compound procedures to arguments:
		○ Evaluate body of procedure with each formal parameter replaced by corresponding argument .

The substitution model is a simplified and incomplete model of procedure application. Its purpose is to help beginners think about procedure application.

Typical interpreters do not evaluate procedure applications by manipulating the text of a procedure to substitute values for the formal parameters. In practice, the “substitution” is accomplished by using a local environment for the formal parameters. 

Normal Order:
	- substitute operand expressions for parameters until expression with only primitive operators
	- then perform evaluation
	- fully expand then reduce
Applicative Order
	- evaluates operator and operands, then applies procedure to arguments
	- evaluate then apply
